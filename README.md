# Norsk Nettverk — v2 (Mapping App)

Mobile-first investigative tool exploring relationships between Norwegian politicians, board members, and corporate roles. **Version 2** of [`froand/norwegian-board-network`](https://github.com/froand/norwegian-board-network), rebuilt around the Google Stitch "Nordic Nexus" mobile concept.

## Status

🚀 **Live demo** — [norsk-nettverk-v2-web.salmonocean-8b8be80d.swedencentral.azurecontainerapps.io](https://norsk-nettverk-v2-web.salmonocean-8b8be80d.swedencentral.azurecontainerapps.io)

v1 continues to run independently at the original repo.

## Stack

- **Frontend**: Next.js 16 (App Router) + TypeScript + Tailwind CSS v4 + PWA, Next.js standalone output
- **Backend**: Node.js + Express + TypeScript (forked from v1)
- **Hosting**: Azure Container Apps in `norsk-nettverk-v2` (Sweden Central)
- **CI/CD**: GitHub Actions (OIDC) → Azure Container Registry → Container Apps
- **Photos**: Wikidata SPARQL (P18) cached to `backend/data/photos.json`, Stortinget personbilde where present

## Design

The dark "Nordic Nexus" theme uses Material 3 tokens (cyan primary, gold tertiary, dark navy surfaces). See `frontend/src/app/globals.css` for the design tokens.

Four-tab mobile nav:

- **Dashboard** — Conflict score overview, Revolving Door feed, sector watchlist
- **Nettverk** — Force-directed graph (touch-optimized)
- **Katalog** — Searchable directory with badges (REVOLVING DOOR RISK / CLEAN PROFILE / DISCLOSURE NEEDED)
- **Varsler** — Live investigative feed + personal watchlist

## Data sources

Inherits all v1 data sources: Brønnøysund (Brreg), Stortinget, Regjeringen, Karantenenemnda PDFs, and the curated person/party seed dataset.

## Refreshing person photos

The photo cache lives in `backend/data/photos.json` and is generated by a one-shot Wikidata SPARQL fetcher. Re-run it if the person list grows or after ~60 days:

```pwsh
cd backend
npx tsx src/scripts/fetch-photos.ts
```

Then commit `backend/data/photos.json`. Portrait precedence at runtime: Stortinget personbilde (for sitting MPs) → Wikimedia Commons (via Wikidata `P18`) → initials disc fallback.

## Azure deployment

The app runs in resource group **`norsk-nettverk-v2`** (Sweden Central) with two Container Apps and one shared user-assigned identity:

| Resource | Name | Purpose |
| --- | --- | --- |
| Container Registry | `norsknettverkv2acr` | Holds backend + frontend images. Admin disabled; pulls use the managed identity below. |
| Managed identity | `norsk-nettverk-v2-pull-id` | Granted `AcrPull` on the ACR so both apps can pull without registry credentials. |
| Container Apps env | `norsk-nettverk-v2-env` | Shared compute env (consumption profile). |
| Backend app | `norsk-nettverk-v2-api` | **Internal ingress only** on port 3011. Reachable from the frontend over Container Apps internal DNS — never public. |
| Frontend app | `norsk-nettverk-v2-web` | **Public ingress** on port 3000. Calls the backend server-side via `API_URL` injected from Bicep. |
| Log Analytics | `norsk-nettverk-v2-logs` | 30-day retention, PerGB2018. |
| App Insights | `norsk-nettverk-v2-appinsights` | Workspace-backed, connected to backend via `APPLICATIONINSIGHTS_CONNECTION_STRING`. |

Because all backend calls happen in React Server Components, the browser never talks to the backend directly — so no `NEXT_PUBLIC_API_URL` is baked into the bundle and no CORS exposure is needed.

### One-time bootstrap

```pwsh
# 1. Create RG + all resources (placeholder images on first run)
az group create -n norsk-nettverk-v2 -l swedencentral
az deployment group create -g norsk-nettverk-v2 --template-file infra/main.bicep

# 2. Push real images
az acr build --registry norsknettverkv2acr --image norsk-nettverk-v2-backend:v1  --file backend/Dockerfile  backend/
az acr build --registry norsknettverkv2acr --image norsk-nettverk-v2-frontend:v1 --file frontend/Dockerfile frontend/

# 3. Roll the apps to the real images
az containerapp update -g norsk-nettverk-v2 -n norsk-nettverk-v2-api `
  --image norsknettverkv2acr.azurecr.io/norsk-nettverk-v2-backend:v1
az containerapp update -g norsk-nettverk-v2 -n norsk-nettverk-v2-web `
  --image norsknettverkv2acr.azurecr.io/norsk-nettverk-v2-frontend:v1
```

### Continuous deploy from GitHub Actions

`.github/workflows/deploy.yml` builds whichever side changed (path-filtered) and rolls the matching Container App to a new revision tagged `v<timestamp>-<shortsha>`. Auth uses OIDC — no long-lived service principal secrets.

**Required repository secrets** (already configured for this repo):

| Secret | Value |
| --- | --- |
| `AZURE_CLIENT_ID` | App registration `norsk-nettverk-v2-gh-deploy` (federated to `repo:froand/Norsk-Nettverk-mapping-App:ref:refs/heads/master`) |
| `AZURE_TENANT_ID` | Tenant id |
| `AZURE_SUBSCRIPTION_ID` | Subscription id |

**Required repository variables**:

| Variable | Value |
| --- | --- |
| `AZURE_RG` | `norsk-nettverk-v2` |
| `AZURE_ACR` | `norsknettverkv2acr` |
| `BACKEND_APP` | `norsk-nettverk-v2-api` |
| `FRONTEND_APP` | `norsk-nettverk-v2-web` |
| `FRONTEND_URL` | Public frontend URL used by the post-deploy smoke test |

### Recreating the federated identity in another tenant

```pwsh
$repo = "froand/Norsk-Nettverk-mapping-App"
$app  = az ad app create --display-name "norsk-nettverk-v2-gh-deploy" --query appId -o tsv
az ad sp create --id $app | Out-Null

$cred = @{
  name      = "gh-master"
  issuer    = "https://token.actions.githubusercontent.com"
  subject   = "repo:${repo}:ref:refs/heads/master"
  audiences = @("api://AzureADTokenExchange")
} | ConvertTo-Json -Compress
$cred | Out-File -Encoding utf8 cred.json
az ad app federated-credential create --id $app --parameters "@cred.json"

$sub = az account show --query id -o tsv
az role assignment create --assignee $app --role "Contributor" `
  --scope "/subscriptions/$sub/resourceGroups/norsk-nettverk-v2"
az role assignment create --assignee $app --role "AcrPush" `
  --scope "/subscriptions/$sub/resourceGroups/norsk-nettverk-v2/providers/Microsoft.ContainerRegistry/registries/norsknettverkv2acr"
```
